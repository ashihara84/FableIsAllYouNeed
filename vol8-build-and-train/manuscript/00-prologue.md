# 序章 最後のラスボスと、道具の解禁

## 0.1 第7巻の到達点 — 部品は全部ある

第7巻の終章で、私たちは1枚の表を完成させました。論文 Section 3 の式番号と、自分のコードの行が、1対1で握手している対応表です。式(1)の attention、multi-head のテンソル整形、position-wise FFN、positional encoding——図1に描かれたすべての箱に、単体テスト済みの自分の実装が対応しています。`python3` で実行すれば、assert は全部通ります。

あの表を、私たちは「卒業証書の半分」と呼びました。

なぜ半分なのか。第7巻終章 9.4 で正直に数えたとおり、まだやっていないことが2つ残っているからです。

**1つ目、組み立てていません。** 部品の単体テストは、あくまで単体の保証です。embedding から出口の softmax まで、encoder 6段と decoder 6段を1本の動くプログラムにつなぐ作業は、まだ何もしていません。

**2つ目、訓練していません。** こちらが本命です。これまでのすべての巻を通して、Transformer の重み $W$ は乱数のままでした。テストが確かめたのは「配管が正しい」ことであって、「良い翻訳をする」ことではありません。完璧に組み上がった Transformer も、乱数の $W$ のままなら、でたらめな文を吐きます。

ここで、この巻全体を貫く主題を立てておきます。

**部品が動くことと、全体が学習することは、別問題です。**

全部品の単体テストが通っていても、数百万個のパラメータを持つ全体が安定して学習する保証は、どこにもありません。学習率が大きすぎれば loss は発散します。mask が1か所漏れていれば、モデルは未来をカンニングして、検証では使いものにならない「実力」を覚えます。論文の Section 5 が、optimizer の係数や warmup のステップ数を小数点以下までわざわざ指定しているのは、飾りではありません。それは、この規模のモデルを学習させるために著者たちが見つけた**処方箋**です。

部品を組み上げ、その処方箋を1行ずつ読み解き、自分の手で実行する。それがこの最終巻の仕事です。

## 0.2 ラスボスの掲示 — 最後の未読領域

では、最後のラスボスと対面しましょう。これまでの巻のラスボスは「1つの式」や「1つのセクション」でした。今回は、論文に残された未読領域の全部——Section 5 Training と Section 6 Results です。その中から3つの文を掲げます。

> *"We trained our models on one machine with 8 NVIDIA P100 GPUs."*
> — Vaswani et al., "Attention Is All You Need", Section 5.2
>
> *"We used the Adam optimizer with β1 = 0.9, β2 = 0.98 and ε = 10^{−9}."*
> — 同論文, Section 5.3
>
> *"On the WMT 2014 English-to-German translation task, the big transformer model (Transformer (big) in Table 2) outperforms the best previously reported models (including ensembles) by more than 2.0 BLEU, establishing a new state-of-the-art BLEU score of 28.4."*
> — 同論文, Section 6.1

訳しておきます。1文目:「私たちは 8 枚の NVIDIA P100 GPU を積んだ1台のマシンでモデルを訓練した」。2文目:「Adam optimizer を $\beta_1 = 0.9$、$\beta_2 = 0.98$、$\epsilon = 10^{-9}$ で使用した」。3文目:「WMT 2014 英独翻訳タスクにおいて、big モデル(Table 2 の Transformer (big))は、アンサンブルを含む既報の最良モデルを 2.0 BLEU 以上上回り、28.4 という新たな最高 BLEU スコアを打ち立てた」。

7巻分の旅を経たいまのあなたなら、単語そのものはかなり読めるはずです。GPU は知っている。Adam という名前は、第2巻終章で「SGD の改良版がある」と予告されたものです。それでも、文として読めるかというと、まだ読めないはずです。

なぜ $\beta_1$ と $\beta_2$ という**2つ**の係数があるのか。$\beta_2 = 0.98$ という値は、何に対して、なぜその値なのか($0.999$ ではいけないのでしょうか)。$\epsilon = 10^{-9}$ などという小さな数は、いったいどこに足すのか。BLEU とは何の点数で、28.4 は 100 点満点なのか。"including ensembles" がわざわざ括弧書きされているのは、何を自慢しているのか。そして 8 枚の P100——この1文は読めた気がするかもしれませんが、本当の意味は「この訓練は、それだけの計算を要求する」という警告です。この警告が私たち自身に降りかかってくることを、第1章で数字として体験します。

ほかにも未読の記号が残っています。Section 5.4 の label smoothing の $\epsilon_{ls} = 0.1$——第4巻終章で「第8巻で回収します」と予告した、シリーズ最後の伏線の1つです。Section 5.3 の warmup の式——第2巻で直観だけ作って、実装を持ち越したものです。

そして、約束します。**この巻を終えると、論文の残り全部——Section 5、Section 6、そして Section 7 の結論まで——が読めるようになります。** 残りの行は、もうありません。終章では、アブストラクトから結論まで、論文全体を頭から通読します。それがこのシリーズの卒業式です。

ただし、今回の約束はそれだけではありません。これまでの巻のゴールは「読める」でした。この巻のゴールはその先にあります。**読めるだけでなく、あなた自身の Transformer が、あなたの手元で動いて、学習している。** loss 曲線が下がっていくのを自分の目で見て、訓練済みの自分のモデルに文を訳させて、その出来を BLEU で測る。論文の Section 5・6 に書かれている営みを、規模を縮めて、全部自分でやる。読了と完走を同時に迎えるのが、この最終巻です。

## 0.3 道具の解禁 — 「作ったから、使う資格がある」

ここで、シリーズ全体に関わる大きな方針転換を宣言します。

**この巻から、PyTorch を解禁します。**

第1巻からここまで、私たちの道具は NumPy と、第5巻で自作した autograd だけでした。世の中に PyTorch という、自動微分もGPU計算も最初から備えたライブラリがあることを知りながら、7巻のあいだ一度も使いませんでした。なぜでしょうか。

理由は、このシリーズの目的そのものです。`loss.backward()` と1行書けば勾配が出てくる世界から始めていたら、backward が**何をしているのか**を知らないまま先へ進めてしまいます。それでは「論文を読んで、自分で実装できる」というゴールに届きません。だから私たちは、微分を手で書き(第2巻)、計算グラフを自作し(第5巻)、attention を NumPy で組みました(第7巻)。ブラックボックスを1つも許さないこと——それがここまでの規律でした。

では、なぜいま解禁するのでしょうか。「最後だからご褒美に」ではありません。シリーズの原則どおり、**需要駆動**です。新しい道具は、困ってから出す。そして PyTorch に対する需要が発生する瞬間は、まさにこの巻にあります。

第1章で、第7巻の部品をそのまま使って、フル Transformer を自作スタックのまま組み上げます。結合テストも通します。**自作スタックでも、Transformer 全体が正しく動くことを、先に証明します。**そのうえで、訓練1ステップの所要時間を実測し、本格的な訓練に必要な総ステップ数を掛け算します。結論だけ先に言えば——**物理的に終わりません**。日や週ではきかない見積もりが出ます。自作 autograd は理解のための道具としては完璧でしたが、GPU で行列積を並列実行する機能を持たないからです。

ここで初めて、「速い実行エンジンが要る」という需要が生まれます。これより前のどの巻にも、この需要はありませんでした。第7巻までの計算は、すべて CPU の NumPy で数秒以内に終わっていたからです。道具は、必要になってから。PyTorch の出番は、シリーズの設計上、ここしかなかったのです。

そしてもう1つ。この7巻分の回り道は、無駄ではありませんでした。むしろ逆です。これから PyTorch の `loss.backward()` を呼ぶとき、あなたはその内側で何が起きているかを、自作した者として知っています。`optim.Adam` を使うとき、その更新式を自分で書いた経験を持っています(第4章で実際に書きます)。ブラックボックスは、中身を知っている人にとってはブラックボックスではありません。ただの、速くて検証済みの部品です。**作ったから、使う資格がある。**これがこの巻の解禁の論理です。

運用のルールを、先に明確にしておきます。

1. **第1章までは自作スタック(NumPy + 第5巻 autograd)でやり切ります。** 組み立て・結合テスト・限界の実測まで。自作でも動くことを証明してから卒業します。
2. **第2章以降の訓練本番は、PyTorch 一本です。並走はしません。** 「同じことを自作版と PyTorch 版の両方で書いて見比べる」ことは、以後やりません。二重実装の維持コストは「論文が読める」というゴールに1ミリも寄与しないからです。最短測地線の原則は、最終巻でも曲げません。
3. **橋渡しは2つだけ。** 第1章の終わり(1.5)に「自作 API ↔ PyTorch API 対応表」を1ページ置きます。自作の `Value` と `backward()` が PyTorch の tensor と `loss.backward()` に、自作の linear が `nn.Linear` に、それぞれどう対応するか。以後の章で PyTorch の API が初登場するたびに、「自作版のどれにあたるか」を脚注で示します。もう1つの橋が、第4章の Adam です。**Adam だけは、フルスクラッチで自作し、PyTorch の `optim.Adam` と挙動を数値で照合します。**自作との並走はシリーズ全体でここが唯一の例外です。optimizer は訓練の成否を握る心臓部であり、ここだけはブラックボックスのまま通過させません。
4. **PyTorch の完成品モジュールは使いません。** `nn.Transformer` も `nn.MultiheadAttention` も禁止です。モデル本体は、素のテンソル演算と最小限の部品(`nn.Linear` 程度)で書き直します。完成品を呼んだ瞬間、第7巻の全部が無意味になるからです。私たちが PyTorch から借りるのは「速い実行エンジンと検証済みの autograd」であって、「出来合いの Transformer」ではありません。

## 0.4 この巻で扱わないこと

最終巻にも、いつもどおり「扱わないこと」を先に白状しておきます。今回は2つです。

**1つ目、大規模化の技術は扱いません。** 複数 GPU にモデルやデータを分割する分散学習(distributed training)、計算を16ビット精度に落として高速化する混合精度(mixed precision)訓練、その他の大規模訓練のエンジニアリングは、すべて省きます。これらは現代の LLM 訓練の現場では必須の技術ですが、**Transformer が学習する原理**の理解には1つも必要ありません。論文の Section 5 を読むのに必要なのは、Adam と warmup と label smoothing がなぜ要るのかであって、GPU を8枚束ねる配線図ではないからです。

**2つ目、WMT 級の本番再現はしません。** 論文は450万文対の英独コーパスで、8枚の P100 を12時間(big モデルなら3.5日)回しています。これを忠実に再現するのは、計算資源的に不可能です——研究機関でない私たちには、現実的な選択肢ではありません。

その代わり、この巻の戦略をはっきり言葉にしておきます。**「論文と同じ形、千分の一の規模」**です。encoder-decoder の構造、BPE、Adam + warmup、label smoothing、beam search——論文の処方箋の**形**は1つも省かず、すべて実装します。縮めるのは規模だけです。パラメータ数は数百万〜数千万、語彙は数千〜1万、訓練時間は GPU 1枚で数十分〜数時間。これがこの巻の標準装備です(GPU がない読者のために、さらに千分の一の極小構成でも同じ形で動く逃げ道を、第2章の脚注で用意します)。

規模を千分の一にしても、学習という現象の骨格は変わりません。loss は同じ理由で下がり、warmup を切れば同じ理由で壊れ、mask が漏れれば同じ症状が出ます。形を全部持っているからこそ、Section 6 の Table 2 を読むとき「論文のモデルはうちの何倍か」という**自分の基準点**を持って読めるのです。
