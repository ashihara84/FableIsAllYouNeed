# 第2章 データ準備 — コーパスからバッチへ

第1章で、Transformer が組み上がりました。第7巻の部品を継ぎ、結合テストを通し、1バッチの丸暗記までやってみせた。しかしそのあとで、私たちは冷たい数字を突きつけられました。自作スタックの1ステップの所要時間から見積もると、本格的な訓練は**物理的に終わらない**。ここからは、序章 0.3 で宣言したとおり PyTorch 一本です。

Sentences were encoded using byte-pair encoding, which has a shared source-target vocabulary of about 37000 tokens. (...) Sentence pairs were batched together by approximate sequence length. ## 2.1 タスクとコーパスの選定: 小規模対訳(または文字レベル翻訳タスク)。「論文と同じ形、千分の一の規模」

最初に決めるべきは、何を訓練するかです。

論文のタスクは英独翻訳でした。訓練データは約450万文対、語彙は共有 BPE で約37000トークン、そして base モデルの訓練には P100 GPU を **8枚並べて12時間**かけています(Section 5.2)。序章 0.4 で宣言したとおり、これをそのまま再現することは、私たちの計算資源では不可能です。

そこで本巻の方針は、一貫してこうです——**論文と同じ形、千分の一の規模**。

「形」とは何か、列挙しておきます。encoder-decoder で対訳を学ぶこと。原言語と目的言語で語彙を共有した BPE。似た長さの文同士で組むバッチ。padding と mask。Adam + warmup(第4章)、label smoothing(第3章)。**これらは全部、論文のまま採用します。** 削るのは規模——データ量、語彙サイズ、パラメータ数、訓練時間——だけです。計算環境は GPU 1枚を前提とし、第5章の訓練本番は「GPU 1枚で数十分〜数時間で終わる」を上限に設計します。パラメータにして数百万〜数千万、BPE 語彙にして数千〜1万程度が、その上限に収まる規模です。

> **CPU しかない読者へ**: 逃げ道があります。規模のつまみをさらに千分の一に絞る——層数と幅を最小にし、トークン化をほぼ文字レベルに落とす——だけで、**同じ形のまま** CPU でも訓練が終わります。トークン化側のつまみはこの章に埋まっています。`data.py` の `NUM_MERGES` を 0 に近づければ BPE は文字単位トークナイザに退化するのでした(第6巻2章: 文字単位は BPE の特殊例)。モデル側のつまみは第5章で示します。

ところで、紙面で動かす実行例には別の要請がもうひとつあります。**結果が決定的で、誰の手元でも一字一句再現できること**です。本物の対訳データセットはダウンロードが要り、版が変わり、ライセンスの確認も要ります。そこで紙面の実行例には、コードの中に直接書き込んだ自作のトイコーパスを使います。規模のはしごを表にします。

|---|---|---|---|
| タスク | 英独翻訳 | 小規模対訳 | 英日翻訳(自作) |
| 対訳ペア数 | 約450万 | 数万〜 | **250** |

トイコーパスは英日の対訳250ペアです。挨拶などの定型文、数の読み、そして「主語 × 動詞 × 目的語」のテンプレートから機械的に作った日常文。すべてこの本のための自作なので、著作権の心配はありません。`data.py` の `make_corpus` の中身はこうです(定型文16ペアと数10ペアのリストは長いので先頭だけ再掲します——全文はファイルにあります)。

```python
def make_corpus():
    """英日のトイ対訳コーパスを返す。完全に決定的(乱数なし)。

    文はすべて本書のための自作で、著作権フリー。日本語は BPE(空白で単語に
    切る)に合わせて分かち書きにしてある。意味の自然さより形の規則性を
    優先した人工対訳である。
    """
    pairs = [
        # --- 挨拶・定型文 ---------------------------------------------------
        ("hello", "こんにちは"),
        ("good morning", "おはよう ございます"),
        ("thank you", "ありがとう ございます"),
        # (... 同様の定型文が計16ペア ...)
        # --- 数 --------------------------------------------------------------
        ("one", "いち"), ("two", "に"), ("three", "さん"),
        # (... "ten" まで計10ペア ...)
    ]

    # --- 日常文(主語 × 動詞 × 目的語のテンプレート)-------------------------
    subjects = [("i", "わたし"), ("you", "あなた"), ("he", "かれ"),
                ("she", "かのじょ"), ("we", "わたしたち"), ("they", "かれら")]
    # (基本形, 三単現, 日本語の述語の型)。{} に目的語が入る
    verbs = [("like", "likes", "{} が すき です"),
             ("eat",  "eats",  "{} を たべます"),
             ("see",  "sees",  "{} を みます"),
             ("want", "wants", "{} が ほしい です")]
    objects = [("apples", "りんご"), ("oranges", "みかん"),
               ("books", "ほん"), ("cats", "ねこ"),
               ("dogs", "いぬ"), ("fish", "さかな"),
               ("tea", "おちゃ"), ("eggs", "たまご")]

    for s_en, s_ja in subjects:
        for v_base, v_3rd, v_ja in verbs:
            v_en = v_3rd if s_en in ("he", "she") else v_base
            for o_en, o_ja in objects:
                pairs.append(("%s %s %s" % (s_en, v_en, o_en),
                              "%s は %s" % (s_ja, v_ja.format(o_ja))))

    # --- 長めの文(目的語2つ。長さの分布に幅を持たせる — 2.3節で効く)--------
    object_pairs = [(objects[0], objects[1]), (objects[2], objects[3]),
                    (objects[4], objects[5]), (objects[6], objects[7])]
    for s_en, s_ja in subjects[:2]:                       # i, you のみ
        for v_base, _, v_ja in verbs:
            for (o1_en, o1_ja), (o2_en, o2_ja) in object_pairs:
                pairs.append(
                    ("%s %s %s and %s" % (s_en, v_base, o1_en, o2_en),
                     "%s は %s" % (s_ja, v_ja.format(o1_ja + " と " + o2_ja))))

    return pairs
```

定型文16 + 数10 + テンプレート文 6×4×8 = 192 + 長文 2×4×4 = 32、合計250ペアです。設計上の判断を3つ、言葉にしておきます。

第一に、**日本語は分かち書き**(単語の間に空白を入れる書き方)にしてあります。第6巻2章の `bpe.py` は `text.split()` で——つまり空白で——単語に切ってからマージを学ぶ実装でした。日本語には本来空白がないので、そのままでは1文がまるごと1単語扱いになってしまいます。本物の日本語処理ならここに前段の分割器を置きますが、最短測地線から外れるので、コーパスの側を最初から切っておきました。

第二に、**「he sees eggs」のような少し不自然な文も許しています**。テンプレートの直積で作る以上、全組み合わせは自然になりません。このコーパスの役割は文学ではなく、対訳という形のデータでパイプライン全体を貫通させることです。

第三に、**わざと長さの違う文を混ぜてあります**。1語の "yes" から "you want tea and eggs" まで。何のためかは、2.3節でわかります。

次に、論文 5.1 の "shared source-target vocabulary" です。第6巻2.5で読んだとおり、これは「英語用とドイツ語用に語彙表を2冊作るのではなく、**両言語のコーパスを混ぜて BPE を1回だけ訓練した**」という意味でした。読んだとおりに、やります。英文と和文を全部つないだ1本のテキストを `train_bpe` に渡す——それだけです。

```python
def _build_vocab():
    """コーパス全文(英 + 日を合算)で BPE を1回だけ訓練し、語彙表を作る。

    英日を混ぜて訓練するのは、論文 5.1 の "shared source-target vocabulary"
    の縮小再現(2.2節)。語彙番号 0, 1, 2 は特殊トークンに予約する。
    """
    corpus = make_corpus()
    text = " ".join(s + " " + t for s, t in corpus)
    merges, vocab = bpe.train_bpe(text, NUM_MERGES)
    # マージの産物は経路違いで重複しうるので、順序を保って一意化してから番号を振る
    seen, tokens = set(), []
    for t in vocab:
        if t not in seen:
            seen.add(t)
            tokens.append(t)
    itos = SPECIALS + tokens            # id -> トークン文字列
    stoi = {t: i for i, t in enumerate(itos)}   # トークン文字列 -> id
    return corpus, merges, stoi, itos


CORPUS, MERGES, stoi, itos = _build_vocab()   # import 時に一度だけ走る(数十ms)
vocab_size = len(itos)
```

ここで使っている定数が3つあります。ファイルの上の方で、こう定義してあります。

```python
# --- 特殊トークン(契約: 第3・5・6章がこの番号を前提にする)------------------
PAD = 0   # 埋め草。バッチを長方形にするための詰め物(2.3節・2.4節)
BOS = 1   # 生成開始の合図。decoder の最初の入力(第6巻6章)
EOS = 2   # 生成終了の宣言。decoder が最後に出すべき記号(第6巻6章)
SPECIALS = ["<pad>", "<bos>", "<eos>"]

NUM_MERGES = 200  # BPE のマージ回数 = 語彙サイズのつまみ(第6巻2章)
```

語彙番号の 0, 1, 2 を、コーパスのどの文にも現れない3つの**特殊トークン**(special token)のために先に予約し、BPE が育てたトークンには 3 番以降を割り振っています。3人の身元と「なぜこの番号か」は 2.4節でまとめて説明します。いまは語彙表の先頭に席が3つ確保されている、とだけ覚えてください。

語彙が確定したら、文を ID 列にする関数と、戻す関数です。

```python
def encode_pair(src, tgt):
    """対訳ペア(文字列2つ)を ID 列2つにする。

    規約(2.4節): src_ids = [本文..., EOS]、tgt_ids = [BOS, 本文..., EOS]。
    decoder 入力と正解への「1トークンずらし」は第3章が tgt_ids から作る。
    """
    src_ids = [stoi[t] for t in bpe.encode(src, MERGES)] + [EOS]
    tgt_ids = [BOS] + [stoi[t] for t in bpe.encode(tgt, MERGES)] + [EOS]
    return src_ids, tgt_ids


def decode(ids):
    """ID 列を文字列に戻す。特殊トークン(PAD/BOS/EOS)は読み飛ばす。"""
    tokens = [itos[int(i)] for i in ids if int(i) >= len(SPECIALS)]
    return bpe.decode(tokens)
```

中身は第6巻の `bpe.encode` / `bpe.decode` に薄い皮をかぶせただけです。皮がやっているのは、トークン文字列と整数 ID の変換(`stoi` / `itos`)と、特殊トークンの付け外し。src の末尾に `EOS`、tgt の前後に `BOS` と `EOS` を付ける規約の理由も 2.4節に預けます。

動かしてみましょう。`data.py` を実行すると(`if __name__ == "__main__":` 以下に検証が入っています)、まず規模が出ます。

語彙サイズ 275 の内訳は、特殊トークン3 + 基礎文字72種(英字・かな)+ マージ産物200。「語彙サイズ = 基礎文字数 + マージ回数」という第6巻2章の会計が、英日混合でもそのまま成り立っています。論文の 37000 がそうだったように、275 も観測値ではなく `NUM_MERGES` という**つまみの設定値**です。

## 2.3 [コード] バッチング: 長さでまとめる(論文5.1の "batched together by approximate sequence length" を実装で回収)、padding と mask の生成

文が ID 列になりました。あとはミニバッチ(第3巻5章)に切るだけ——なのですが、第3巻では出てこなかった障害物があります。

ミニバッチは、`(B, L)` の形をした1つの行列として一斉に計算したいのでした(`B` はバッチ内の文の数、`L` は文の長さ)。行列は長方形です。ところが文の長さはまちまちです。"yes" は2トークン、"you want tea and eggs" は6トークン。**長さの違う ID 列を束ねても、長方形になりません。**

解決策は工作と同じで、足りないところに詰め物をします。バッチ内でいちばん長い文に合わせて、短い文の末尾を `PAD` という埋め草トークンで埋めるのです。

```python
def pad_block(seqs):
    """長さの違う ID 列のリストを、PAD で埋めた長方形 (B, L_max) にする。"""
    L = max(len(s) for s in seqs)
    out = np.full((len(seqs), L), PAD, dtype=np.int64)  # PAD=0 なので zeros と同じ
    for k, s in enumerate(seqs):
        out[k, :len(s)] = s
    return out
```

これで長方形は手に入りますが、タダではありません。埋め草のセルにも、行列演算は等しく計算を支払います。`(B, L_max)` の行列のうち PAD が占める割合を、この章では **padding の無駄率**と呼ぶことにします。無駄率 30% なら、GPU の仕事の3割が詰め物の計算に消えるということです。

ここでようやく、第3巻5章の演習問4が回収できます。あのとき「論文はなぜ似た長さの文同士でまとめたのか?」という問いに、「行列は長方形だから、長さの近い文を束ねれば詰め物が減るはず」と**推測**だけして終わっていました。今日は実装して、無駄率を実測します。

```python
def make_batches(pairs, batch_size, rng, by_length=True):
    """encode_pair 済みのペア列をバッチに切る。返り値: (バッチ列, padding率)。

    by_length=True のとき、論文 5.1 の "batched together by approximate
    sequence length" を再現する: シャッフル → 長さで安定ソート(同じ長さの
    中の順序はエポックごとに変わる)→ batch_size ごとに切る → バッチの
    並び順をシャッフル(短い文ばかりの時間帯を作らないため)。
    padding率 = 全バッチの全セルに占める PAD の割合(src と tgt の合算)。
    """
    order = list(rng.permutation(len(pairs)))
    if by_length:
        order.sort(key=lambda i: len(pairs[i][0]) + len(pairs[i][1]))
    chunks = [order[k:k + batch_size] for k in range(0, len(order), batch_size)]
    chunks = [chunks[k] for k in rng.permutation(len(chunks))]

    batches, n_pad, n_cells = [], 0, 0
    for idx in chunks:
        src = pad_block([pairs[i][0] for i in idx])
        tgt = pad_block([pairs[i][1] for i in idx])
        batches.append((src, tgt))
        n_pad += int((src == PAD).sum() + (tgt == PAD).sum())
        n_cells += src.size + tgt.size
    return batches, n_pad / n_cells
```

手順を言葉にすると「混ぜてから、長さで並べ、端から `batch_size` 人ずつ切る」です。シャッフルが2か所に仕込んであることに注意してください。最初のシャッフルは、同じ長さの文の中での順番をエポックごとに変えるためです(**安定**ソートなので、シャッフルの効果が同長グループの中に残ります)。最後のシャッフルはバッチの**並び順**を混ぜるため。これを忘れると、エポックの前半は短文ばかり、後半は長文ばかりという偏った順番でデータが流れてしまいます。論文が "approximate sequence length"(**おおよその**系列長)と書いているのはこの感覚です——長さ順は無駄を減らす手段にすぎないので、厳密に守る必要はなく、むしろ乱数で適度に崩しておくのが健全なのです。

では、実測します。長さでまとめることをやめた場合(`by_length=False`: ただ混ぜて端から切る)と比較すると——

```
padding率(batch_size=32): ランダム 26.4% / 長さ順 8.7%
```

ランダムにバッチを組むと、計算の**4分の1以上が埋め草**に消えます。バッチのどこかに長文が1本入るだけで、その長さまで全行が水増しされるからです。長さ順にまとめると無駄率は 8.7%——**3分の1**になりました。この数字は `rng = np.random.default_rng(42)` のもとで決定的で、`data.py` の assert が固定しています。

```python
    assert rate_len < rate_rand / 2       # 長さ順は無駄を半分以下にする
    assert round(rate_rand, 3) == 0.264   # 本文の数字(rng=42 で固定)
    assert round(rate_len, 3) == 0.087
```

トイコーパスでもこれだけ差が出るのですから、論文の規模——1バッチに約25000トークン、それを10万ステップ——では、この差が GPU 何百時間ぶんにも化けます。"batched together by approximate sequence length" という一句は飾りではなく、**計算費の節約術**だったわけです。なお、論文のバッチは「文の数」ではなく「トークンの総数」を揃えています(approximately 25000 source tokens)。長さ順に並べてから切る以上、短文のバッチは文の数を増やせる——同じ発想のもう一歩先の最適化ですが、私たちの規模では効果が薄いので、本書は `batch_size` 文ずつの単純な形を採ります。

埋め草の後始末が、もうひとつ残っています。PAD のセルは行列の中に居座っている以上、**計算には参加してしまいます**。せめてモデルの意味の側では完全に無視させなければなりません。attention が PAD の位置を**見ない**ようにする——第7巻3章で作った padding mask の出番です。

```python
def make_pad_mask(batch):
    """padding mask (B, L)。True = 本物のトークン、False = PAD。

    第7巻3章の「True = 見てよい」規約に合わせてある。attention の scores
    (B, n, m) に掛けるときは mask[:, None, :] と整形してブロードキャストする
    (全 query 行に同じ禁止が掛かる — 第7巻3章 padding_mask と同じ形)。
    """
    return batch != PAD
```

たった1行ですが、規約がすべてです。第7巻3章の `attention(Q, K, V, mask)` は「**True = 見てよい**」という約束の bool 配列を受け取るのでした。`make_pad_mask` はその約束に合わせて、本物のトークンの位置だけ True を返します。本当に末端まで効くのか——この mask を第7巻の `attention` に差し込むと PAD への注意の重みが**厳密に 0** になるのか——は、検証スクリプト `verify_data.py` が確認しています。核心部分を抜き出します(ファイルには本章の契約全体の検査が入っています)。

```python
src, _ = batches[0]                       # (B, m)
B, m = src.shape
mask = data.make_pad_mask(src)            # (B, m)  True = 見てよい

rng = np.random.default_rng(0)
n, d_k = 4, 8
Q = rng.standard_normal((B, n, d_k))
K = rng.standard_normal((B, m, d_k))
V = rng.standard_normal((B, m, d_k))
_, w_masked = attention(Q, K, V, mask=mask[:, None, :])   # (B, n, m)
_, w_open = attention(Q, K, V)

pad_pos = np.broadcast_to(~mask[:, None, :], (B, n, m))   # PAD の位置(全 query 行)
assert (w_masked[pad_pos] == 0.0).all()                   # mask あり: 重みは厳密に 0
assert np.allclose(w_masked.sum(axis=-1), 1.0)            # 本物だけで和が 1
assert (w_open[pad_pos] > 0).all()                        # mask なし: PAD に漏れる
```

mask を渡せば PAD の重みは厳密に 0、渡さなければ正の重みが漏れる——両方向が assert で確認できます。PAD の後始末はもう1か所、**損失計算から PAD を除外する**仕事が残っていますが、それは損失を作る第3章でやります。

## 2.4 特殊トークン: BOS / EOS / PAD の規約

道具がひととおり揃ったので、ここで2.2節から先送りしてきた宿題を片付けます。語彙の先頭に席を予約した3つの特殊トークンは、何者で、なぜ必要なのか。

3人とも、初対面ではありません。

**BOS(beginning of sequence)— 自己回帰の開始合図。** 第6巻6章で、decoder の最初の入力として登場しました。decoder は「直前までの出力を入力して、次の1トークンを出す」自己回帰の機械です。では**最初の**1トークンを出すとき、何を入力すればいいのでしょうか。まだ何も出力していないのだから、入力がありません。この「何もない」を表す専用トークンが BOS です。生成のスタートの号砲であり、訓練時には teacher forcing の入力列の先頭に立ちます(第6巻6.4)。

**EOS(end of sequence)— 終了の宣言。** 同じく第6巻6章から。翻訳の出力が何トークンで終わるかは、事前にわかりません。機械が「ここで終わり」と自分で宣言できるのは、EOS が語彙にあり、訓練データの各文末に置かれ、「最後に EOS を出す」こと自体を学習しているからです。EOS がなければ、生成は永遠に止まれません。

**PAD(padding)— 長方形の埋め草。** これだけが本章の新顔ですが、必要性は第3巻5章で予告済み、実装は2.3節で済んでいます。長さの違う文をひとつの行列に束ねるための詰め物であり、文の意味とは無関係——だからこそ mask(2.3節)と損失からの除外(第3章)で、徹底的に「いない者」として扱われます。

さて、`data.py` の規約を契約書の形で明記します。**この番号と形を、第3・5・6章のコードが前提にします。**

| 項目 | 規約 |
|---|---|
| 番号 | `PAD = 0`、`BOS = 1`、`EOS = 2`。BPE のトークンは 3 番以降 |
| src 側 | `[本文 ..., EOS]`(BOS は付けない) |
| tgt 側 | `[BOS, 本文 ..., EOS]` |
| バッチ | 末尾を PAD で埋めた `(B, L)` の `np.int64` 配列 |
| mask | `make_pad_mask(batch)`、True = 本物(第7巻3章の規約) |

2点、補足します。

まず番号について。第6巻6章では `BOS = 0, EOS = 1` でした(あのミニチュアは全文字列が同じ長さで、PAD が不要でした)。本章で PAD が加わるにあたり、**0 番は PAD に譲る**ことにしました。理由は実利です。`np.zeros` で作った配列がそのまま「全部 PAD」を意味するので、`pad_block` のような埋め草処理が自然に書けます。さらに第3章で使う PyTorch の `nn.Embedding` には `padding_idx=0` という引数があり、「0 番は埋め草」は PyTorch の世界の慣習でもあります。逆に言えば、この割り当ては数ある流儀のひとつにすぎません。特殊トークンの規約に正解はなく、**モジュール全体で一貫していることだけが命**——一貫性を保証するのが `data.py` という1枚の契約書、というわけです。

次に、src と tgt の非対称について。tgt に BOS と EOS が両方付くのは、自己回帰には開始の号砲と終了の宣言が要るからでした。一方 **src に BOS がない**のは、encoder が自己回帰をしないからです。encoder は入力文を一度に全部読む側で、「最初の1トークンを生み出す」局面がそもそもありません。EOS だけは付けて、「入力はここまで」という情報を文自身に持たせます。そして tgt を `[BOS, 本文..., EOS]` の**1本**で持っておくと、第3章で decoder への入力 `tgt[:-1]` と正解 `tgt[1:]` が**1トークンずらすだけ**で切り出せます。第6巻6章の終わりに「decoder への入力は訓練前から全部わかっている——この事実が第8巻3章の訓練ループの前提になる」と予告した、あのずらしです。データ側の準備は、これで整いました。

## まとめ

- 本巻のタスクは小規模対訳。方針は「**論文と同じ形、千分の一の規模**」——共有 BPE・長さ順バッチ・mask・Adam + warmup という形は論文のまま、規模だけを GPU 1枚(数十分〜数時間)に収まるよう絞る。紙面の実行例は決定的な自作英日コーパス250ペア
- BPE は第6巻2章の `bpe.py` を **import して再利用**。英文と和文を混ぜて1回だけ訓練するだけで、論文 5.1 の "shared source-target vocabulary" の縮小再現になる(語彙275 = 特殊3 + 文字72 + マージ200)
- 長さの違う文を長方形に束ねるため **PAD** で埋める。埋め草も計算費を食うので、**似た長さ同士でまとめてから**バッチ化する(論文 5.1 の "batched together by approximate sequence length")。実測で無駄率 26.4% → **8.7%**、3分の1になった
- PAD は mask で attention から隠す(第7巻3章の `attention` に差すと重みが厳密に 0)。損失からの除外は第3章
- 特殊トークンの契約は **PAD=0, BOS=1, EOS=2**。src は `[..., EOS]`、tgt は `[BOS, ..., EOS]`。BOS は自己回帰の開始合図、EOS は終了宣言、PAD は埋め草——tgt の1本持ちが第3章の「1トークンずらし」の仕込みになる

## 演習

**問1(バッチ効率 — padding の無駄率を測る)** `make_batches` は無駄率を返します。`batch_size` を 8, 32, 128, 250 と変え、`by_length=True / False` の両方で無駄率を測って表にしてください(`rng` は毎回 `np.random.default_rng(42)` で作り直すこと)。表から、(1) ランダムバッチの無駄率はバッチを大きくするとどこへ向かうか、(2) 長さ順の利得はバッチサイズとどういう関係にあるか、(3) `batch_size=250`(全データが1バッチ)で2つの方式の無駄率が**完全に一致する**のはなぜか、を説明してください。

<details><summary>略解</summary>

実測(`data.py` のコーパスと rng=42)は次のとおり。

| batch_size | ランダム | 長さ順 |
|---|---|---|
| 8 | 19.0% | 2.3% |
| 32 | 26.4% | 8.7% |
| 128 | 27.0% | 19.2% |
| 250 | 27.0% | 27.0% |

(1) ランダムは大きくするほど悪化し、27% に飽和します。バッチの幅は**バッチ内の最長文**が決めるので、バッチが大きいほど長文を引き当てる確率が上がり、ほぼ毎バッチが最大幅になるからです。(2) 長さ順の利得はバッチが小さいほど大きい(8文なら 2.3%)。小さいほど「同じ長さの文だけで1バッチ」を組みやすいからです。(3) 250文を1バッチにすると、並べ替えても**全文が同じ1つの長方形**に入り、幅はコーパス全体の最長文で決まります。中身の順番は長方形の形に影響しません。長さ順バッチが効くのは「並べる」からではなく、並べたあとで**切る**からです。

</details>

**問2(mask を外すと何が漏れるか)** `verify_data.py` の `w_open`(mask なしの注意の重み)を使って、PAD の位置に漏れている重みの合計を query 行ごとに計算してください(`w_open` のうち `mask` が False の位置の和)。平均で何%程度が PAD に漏れていますか。また、この漏れを放置して訓練すると何がまずいのか、「同じ文でも、どのバッチに入るかで PAD の本数が変わる」という事実を使って説明してください。

<details><summary>略解</summary>

`leak = np.where(mask[:, None, :], 0.0, w_open).sum(axis=-1)` が行ごとの漏れです。先頭バッチ(幅5、PAD は 160 セル中 31)では平均およそ 17〜18% が PAD に流れます(乱数の Q, K では重みが一様に近いので、漏れは PAD のセル比 31/160 ≈ 19% に近い値になります)。放置すると、PAD の位置にも埋め込みベクトル(0 番の行)があるため、attention の出力に「埋め草の意味ベクトル」が混入します。しかも PAD の本数はバッチの組み方で変わるので、**同じ文の表現が、同じパラメータでもバッチ次第で変わる**ことになります。モデルの出力がデータではなくバッチ編成という偶然に依存する——これが mask で漏れを厳密に 0 にしておく理由です。

</details>

**問3(共有語彙は英日で何を買ったか)** 語彙を共有せず、英語側と日本語側で別々に BPE を訓練したらどうなるでしょうか。`bpe.train_bpe` を英文だけ・和文だけでそれぞれ `num_merges=200` で訓練し、(打ち止めまで育てた)語彙サイズの合計を、共有語彙を同様に打ち止め(`num_merges=400`)まで育てたサイズと比べてください。また、英語側と日本語側の語彙に共通して現れるトークンを数えてください。結果から、「論文(英独)で語彙共有が効いた理由」と「英日では事情が違う理由」を述べてください。

<details><summary>略解</summary>

実測では、別々に訓練すると英語側165 + 日本語側168 = 333。共有で打ち止めまで育てると 332 で、ほぼ同じです。両語彙に共通するトークンは、単語末マーカー `</w>` の**たった1個**。英日では語彙を共有しても表は実質1つも縮みません。英語はラテン文字、日本語はかなで、**基礎文字の集合がそもそも交わらない**からです。論文の英独は事情が逆で、同じアルファベットを使い、固有名詞・数字・同根語が同じ綴りで現れるため、共有した部品のぶんだけ語彙が節約できます(第6巻2.5)。では英日で共有が無意味かというと、そうではありません。語彙表が1冊なら embedding 行列と出力側の softmax(第3章)も1組で済み、実装が単純になります。本書が共有を採るのは、節約のためというより**論文と同じ形を保つ**ためです。

</details>
